# 设计决策示例：深色极简跑步表盘

## 原始需求

> 为圆形 AMOLED 跑步手表设计一个深色极简表盘。时间要非常清楚，同时展示心率和电量，
> 支持 Always-On。

## 1. 解析设计意图

```json
{
  "audience": "跑步用户",
  "style": ["深色", "极简", "运动"],
  "informationDensity": "low",
  "mustHaveComponents": ["time", "heartRate", "battery"],
  "priorities": {
    "time": 1,
    "heartRate": 2,
    "battery": 3
  },
  "alwaysOnRequired": true
}
```

## 2. 查询设备能力

结构化查询需要确认：

- 设备是圆形 AMOLED；
- 分辨率和安全区域；
- API Level 是否支持组件所需能力；
- Always-On 需要使用简化低亮度设计；
- 当前产品是否已经实现时间、心率和电量组件。

若目标设备无法提供某组件所需数据，必须在生成候选前告知用户，而不是先设计再失败。

## 3. 选择组件

| 组件 | 决策 | 理由 |
| --- | --- | --- |
| 时间 | 选择，主组件 | 用户明确要求，表盘核心信息 |
| 心率 | 选择，次组件 | 用户明确要求，与跑步场景高度相关 |
| 电量 | 选择，辅助组件 | 用户明确要求，适合小面积状态提示 |
| 步数 | 省略 | 用户要求极简，且未明确要求 |
| 日期 | 省略 | 用户未要求，避免削弱时间层级 |

这里不是让模型自由挑选。必选组件先锁定，可选组件根据极简目标、视觉面积和优先级被过滤。

## 4. 选择布局

检索条件：

- 圆形屏幕；
- 一个主组件、两个次要组件；
- 低信息密度；
- 极简、运动风格。

匹配布局：`center-time-bottom-stats`

选择理由：

- 中心区域适合突出大号时间；
- 底部左右两个槽位适合心率和电量；
- 不需要把元素贴近圆屏边缘；
- Always-On 时可以只保留中心时间。

## 5. 选择组件表现形式

| 组件 | Active 模式 | Always-On 模式 | 理由 |
| --- | --- | --- | --- |
| 时间 | 大号数字 | 简化大号数字 | 始终保持最高可读性 |
| 心率 | 图标 + 数字 | 默认省略或仅数字 | Active 有辨识度，Always-On 降低视觉与功耗负担 |
| 电量 | 数字或小图标 + 数字 | 默认省略或仅数字 | 小面积即可表达状态 |

具体是否在 Always-On 保留心率和电量，需要设备规则、产品能力和用户偏好共同决定。

## 6. 选择颜色

主题基础色：

- 背景：黑色；
- 主色：白色；
- 强调色：适合运动风格的绿色；
- 次要色：灰色。

状态覆盖：

| 组件状态 | 颜色决策 |
| --- | --- |
| 心率正常 | 绿色强调色 |
| 心率过期或不可用 | 灰色，并显示 `--` 或过期提示 |
| 电量正常 | 灰色或白色 |
| 电量低 | 警告色 |
| 电量严重不足 | 危险色 |
| Always-On | 使用低亮度安全色，减少高亮元素 |

因此，颜色不是一次性写死。最终颜色由主题、组件语义、数据状态和显示模式共同决定。

## 7. 生成候选差异

候选 A：

- 中心大号时间；
- 心率和电量位于底部左右；
- 极少装饰。

候选 B：

- 时间略高于中心；
- 底部使用细线分隔的状态栏；
- 强调运动感。

候选 C：

- 中心时间；
- 心率用小型进度环表达；
- 电量使用纯数字。

三个候选必须通过设备、布局和 Always-On 规则检查。若进度环在 Always-On 不合适，则候选
C 的 Always-On 版本必须降级。

## 8. 预览与用户反馈

用户反馈：

> 选择 A，但心率比电量更重要，希望心率更大，并且绿色不要太亮。

系统应只修改：

- 心率字号或槽位面积；
- 心率强调色；
- 必要时调整电量组件以保持布局平衡。

系统不应擅自增加日期、改变时间形式或替换整个布局。

## 10. 进入代码生成的条件

- 用户确认候选 A 的修改版本；
- 所有必选组件均存在；
- 状态与 Always-On 策略完整；
- 自动视觉硬规则通过；
- 组件、布局和样式选择都有决策证据；
- 规格锁定唯一版本。
