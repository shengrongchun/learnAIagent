# 组件视觉变体与风格系统

## 核心问题

“电量组件”只定义数据含义，不等于定义它的展示效果。运动风、科技风、卡通风需要不同的
视觉语言；同一种风格下，也需要多个布局、形状和细节变体。

组件效果不能每次由模型从零自由生成，否则会出现：

- 风格不一致；
- 预览能画但 Monkey C 无法实现；
- 同一表盘中的组件像来自不同产品；
- 批量结果只是随机变化，质量不可控；
- 新生成效果无法稳定复用和测试。

解决方案是建立**可组合、可参数化、经过验证的组件视觉变体库**。

必须承认：没有人工设计基线、合法的高质量案例和持续用户反馈，仅靠 Agent 无法保证视觉
质量。Agent 能提高组合、适配和探索效率，但产品的视觉质量上限由设计系统与认证变体库决定。

## 五层视觉系统

```text
绘制原语
  -> 组件视觉变体
      -> 风格系统
          -> 状态与设备模式覆盖
              -> 有限参数变化
```

### 1. 绘制原语

预览器和 Monkey C 都能实现的最小视觉能力：

- 文本；
- 图标；
- 线条；
- 圆、圆弧和进度环；
- 矩形、圆角矩形；
- 点阵和刻度；
- 简单填充与描边；
- 已验证的位图资源。

每个原语必须同时有：

- Web/SVG 或 Canvas 渲染实现；
- Monkey C 渲染实现或代码模板；
- 支持设备和显示模式；
- 性能和资源成本。

### 2. 组件视觉变体

组件变体是“组件数据如何通过原语表达”的可复用配方。

例如电量组件可以有：

- `battery-text-compact`：纯数字百分比；
- `battery-icon-outline`：轮廓电池图标 + 数字；
- `battery-arc-progress`：圆弧进度；
- `battery-segment-gauge`：分段刻度；
- `battery-capsule-fill`：胶囊填充；
- `battery-character-charge`：卡通角色能量表现。

每个变体明确：

- 使用哪些绘制原语；
- 可调整哪些参数；
- 最小和最大尺寸；
- 支持哪些风格、状态、设备和显示模式；
- 对应预览渲染器与 Monkey C 模板；
- 已通过哪些质量测试。

### 3. 风格系统

风格不是一个标签或一组颜色，而是一套协调的设计令牌和选择规则。

| 风格 | 视觉语言示例 |
| --- | --- |
| 运动风 | 高对比、粗线条、斜切、速度感、清晰数字、少装饰 |
| 科技风 | 细线、分段刻度、网格、冷色、模块化、仪表感 |
| 卡通风 | 圆润轮廓、柔和或鲜艳色彩、角色化图标、夸张状态反馈 |

风格系统应定义：

- 色板和语义色；
- 字体类型、字重和数字形式；
- 线宽、圆角、留白和密度；
- 图标语言；
- 允许和禁止的组件变体；
- 装饰原语；
- Active 与 Always-On 的降级规则。

### 4. 状态与设备模式覆盖

组件选定视觉变体后，还要应用：

- normal、warning、critical、unavailable 等数据状态；
- Active、low-power、Always-On 等设备显示模式；
- 用户目标达成等业务状态。

状态改变颜色、强调程度、可见元素和降级表现，但不能破坏风格一致性。

### 5. 有限参数变化

同一个变体通过受控参数产生丰富效果，例如：

- 圆弧起始角度和长度；
- 分段数量；
- 图标与数字的相对位置；
- 线宽和圆角；
- 强调色选择；
- 数字字号；
- 装饰元素数量；
- 信息密度。

每个参数必须有允许范围和组合约束。模型只能从允许空间中选择或提出变化，不能直接创造
任意绘制逻辑。

## 三种风格下的组件示例

### 电量

| 风格 | 推荐变体 | 展示特点 |
| --- | --- | --- |
| 运动风 | 圆弧进度、粗轮廓图标、紧凑数字 | 高对比、快速读取、低电量强提醒 |
| 科技风 | 分段刻度、细线仪表、模块数字 | 精密、仪表化、冷色强调 |
| 卡通风 | 胶囊填充、角色能量、表情状态 | 亲和、状态变化明显、轮廓圆润 |

### 心率

| 风格 | 推荐变体 | 展示特点 |
| --- | --- | --- |
| 运动风 | 心形图标 + 大数字、脉冲线 | 强调实时数据和可读性 |
| 科技风 | ECG 细线、分段环、模块标签 | 数据仪表感和精密感 |
| 卡通风 | 圆润心形、角色表情、跳动符号 | 情绪化表达，但必须保持数值可读 |

## 丰富性从哪里来

丰富性不依赖无限随机生成，而来自多个受控维度的组合：

```text
组件类型
× 视觉变体
× 风格系统
× 布局槽位
× 参数组合
× 数据状态
× 设备显示模式
× 主题色方案
```

例如同一个运动风电量组件可以有：

- 三种经过验证的视觉变体；
- 每种变体两种图文排列；
- 三种允许的圆弧长度或分段密度；
- 多套运动风主题色；
- normal、low、critical、Always-On 状态。

组合空间足够丰富，但每个结果仍然来自已验证能力。

## 多样性生成策略

生成多个候选时，Agent 先制定差异计划，而不是随机调用模型多次。

示例：

```json
{
  "candidatePlans": [
    {
      "id": "A",
      "layoutFamily": "center-focus",
      "batteryVariant": "battery-arc-progress",
      "heartRateVariant": "heart-rate-icon-number"
    },
    {
      "id": "B",
      "layoutFamily": "dashboard",
      "batteryVariant": "battery-segment-gauge",
      "heartRateVariant": "heart-rate-pulse-line"
    },
    {
      "id": "C",
      "layoutFamily": "asymmetric-sport",
      "batteryVariant": "battery-icon-outline",
      "heartRateVariant": "heart-rate-large-number"
    }
  ]
}
```

候选之间应在核心设计维度上变化，同时共享同一风格系统，避免“差异大但像三个不同产品”。

## 组件变体从哪里来

### 来源 1：人工设计

首批核心变体应由人设计和审核。它们建立产品的质量基线，也是后续生成的可靠素材。

可通过产品负责人自行设计、邀请设计师参与、举行内部设计评审或购买合法设计服务获得。

### 来源 2：优秀案例抽象

从合法、可参考的优秀设计中抽象布局和视觉原则，不能直接复制受版权保护的图形或资源。

可以使用自有作品、获得授权的设计、开放许可证资源和用户明确授权贡献的组件。每个资源
必须记录来源与许可证。

### 来源 3：Agent 提议新变体

Agent 可以基于已有原语和风格规则提出新配方，但新配方默认处于实验状态，不能直接用于
正式输出。

### 来源 4：用户修改沉淀

被用户多次选择的修改可以整理成新变体或参数预设，经审核后加入正式库。

## 正式生成与探索生成分离

### 正式生成路径

- 只使用 `certified` 组件变体和允许参数；
- 目标是稳定、可预览、可实现和可维护；
- 默认用于最终 Monkey C 输出。

### 探索生成路径

- Agent 可以重新组合原语、提出新参数范围或新变体；
- 目标是寻找新的视觉方向；
- 结果必须标记为实验性；
- 未通过完整认证前不能作为正式输出。

这样既不会因为追求稳定而永远只有少量模板，也不会因为追求多样性而让正式结果失控。

## 组件变体生命周期

```text
draft -> preview-tested -> simulator-tested -> human-approved -> certified -> deprecated
```

只有 `certified` 变体默认参与正式候选生成。

新变体进入正式库前必须：

1. 符合风格设计令牌；
2. 通过所有关键状态预览；
3. 通过边界、重叠、对比度和可读性检查；
4. 有对应 Monkey C 实现；
5. 在目标设备模拟器或真机验证；
6. 通过人工视觉审核；
7. 记录来源、版本、适用范围和测试结果。

## 质量保障

### 单组件质量

- 数值或状态是否快速可读；
- 正常、异常、缺失和 Always-On 是否完整；
- 小尺寸下是否仍能识别；
- 是否符合目标风格；
- 预览与 Monkey C 实现是否一致；
- 性能和资源成本是否合格。

### 组件组合质量

单独好看的组件组合后可能很差，因此还要检查：

- 所有组件是否共享同一设计语言；
- 主次层级是否清楚；
- 图标、线宽、圆角和字体是否一致；
- 装饰是否过量；
- 风格强度是否平衡；
- 组件之间是否争抢视觉注意力。

### 批量质量

- 候选有真实差异；
- 候选没有超出认证变体和允许参数；
- 同一候选内部风格一致；
- 不重复近期被拒绝的组合；
- 用户保留率和人工评分达到阈值。

## Agent 的角色

Agent 不是视觉效果的唯一创造者，而是：

- 解析用户想要的风格与感觉；
- 检索适合的认证组件变体；
- 规划候选之间的差异；
- 在允许参数范围内组合和调整；
- 解释为什么选择这些变体；
- 根据用户反馈修改；
- 提议值得进入实验区的新变体。

产品质量由设计系统、认证变体、规则、测试和人工审核共同保障。

持续扩展组件和风格的完整流程见
[持续学习与设计孵化系统](持续学习与设计孵化系统.md)。生产生成和设计探索必须分离。

## 第一版现实范围

不要一开始追求几十种风格和数百个变体。第一版建议：

- 运动风、科技风、卡通风 3 套风格系统；
- 电量、心率、时间 3 个核心组件；
- 每个核心组件每种风格先做 2 个高质量变体；
- 每个变体覆盖 Active、Always-On、normal 和 unavailable；
- 电量额外覆盖 low 和 critical；
- 所有正式变体均有 Web 与 Monkey C 实现。

这已经形成至少 18 个核心视觉变体，并可通过布局、状态、主题参数产生大量候选。先保证
这些变体真正可用，再扩展日期、步数和更多风格。
