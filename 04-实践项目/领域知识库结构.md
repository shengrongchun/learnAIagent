## 各目录的使用方式

| 数据 | 存储形式 | 检索方式 | 原因 |
| --- | --- | --- | --- |
| 设备能力 | JSON | ID、字段和范围过滤 | 必须精确 |
| 组件定义 | JSON | ID、标签和条件过滤 | 必须可解释 |
| 组件视觉变体 | JSON + 渲染实现 | 风格、设备、状态和认证级别过滤 | 保证丰富性与可实现性 |
| 状态样式 | JSON | 状态优先级合并 | 必须确定 |
| 风格系统 | JSON | 风格标签和设计令牌查询 | 保持跨组件一致性 |
| 设计布局 | JSON | 条件过滤 + 语义检索 | 同时需要约束和风格匹配 |
| 硬规则 | JSON/规则函数 | 规则引擎 | 不能由模型决定是否遵守 |
| 官方长文档 | Markdown | 语义检索 + 引用 | 用于解释和补充上下文 |
| 设计案例 | JSON + 预览引用 | 语义检索 + 结构化过滤 | 用于寻找类似成功设计 |
| 用户反馈 | JSONL/数据库 | 聚合、排序和统计 | 用于持续改进选择 |

## 知识条目必须包含的元数据

每条知识至少记录：

- 唯一 ID；
- 版本；
- 类型；
- 来源；
- 适用设备、API Level 或产品版本；
- 生效时间和最后验证时间；
- 置信度；
- 是否为硬规则；
- 负责人或维护方式。

示例：

```json
{
  "id": "amoled-always-on-simplify",
  "version": "1.0",
  "type": "power-rule",
  "severity": "error",
  "source": {
    "type": "official-documentation",
    "url": "https://developer.garmin.com/connect-iq/user-experience-guidelines/watch-faces/"
  },
  "appliesTo": {
    "screenTechnology": "AMOLED",
    "displayMode": "alwaysOn"
  },
  "lastVerifiedAt": "2026-06-09"
}
```

## 检索服务应提供的工具

第一版不需要复杂向量数据库，可以先实现以下 Node.js 工具：

```text
getDeviceProfile(deviceId)
listFeasibleComponents(deviceId, displayMode)
getComponentDefinition(componentId)
listCertifiedVariants(componentId, styleId, deviceId, displayMode)
getComponentStateStyle(componentId, dataState, displayMode)
findLayoutPatterns(shape, componentRoles, styleTags)
evaluateDesignRules(watchFaceSpec)
findSimilarAcceptedDesigns(designIntent)
proposeCandidateDifferencePlan(designIntent, availableVariants)
recordUserDecision(decision)
```

这些工具返回结构化结果，并附上知识条目 ID。Agent 负责组织调用和解释，不能绕过工具直接
声称设备或组件支持某能力。

## 知识合并优先级

当多个知识来源冲突时，按以下优先级处理：

1. 用户明确要求，但不能违反硬规则；
2. 设备与 Garmin 官方硬约束；
3. 当前产品已实现能力；
4. 数据状态与安全降级规则；
5. 设计模式推荐；
6. 历史用户偏好；
7. 模型自身常识。

最低优先级的模型常识不能覆盖任何已检索到的结构化事实或规则。

## 知识更新

- 官方设备和规则变化后，更新对应 JSON 并记录验证日期；
- 新组件必须补齐普通、异常、缺失和 Always-On 状态；
- 新组件视觉变体必须经过预览、Monkey C、设备和人工认证；
- 新布局必须经过人工设计与至少一轮真实预览验收；
- 用户确认和拒绝结果进入反馈库；
- 被多次拒绝的模式降权，而不是直接让模型“记住”；
- 每次知识更新后运行固定评测集。

知识更新由设计孵化流程产生，并通过认证与回归后发布。生产 Agent 对正式知识库只读，不得
根据单次任务直接修改组件、风格、规则或认证状态。

## 评测

领域知识检索单独评测，不与最终代码生成混为一个指标：

| 指标 | 说明 |
| --- | --- |
| 设备事实准确率 | 查询到的设备事实是否正确 |
| 组件可行性判断准确率 | 是否正确过滤不支持组件 |
| 状态样式命中率 | 是否选择正确状态和降级样式 |
| 认证变体使用率 | 正式候选是否只使用允许的认证变体 |
| 风格一致率 | 同一候选中的组件是否属于协调的设计语言 |
| 变体人工接受率 | 推荐变体是否被用户保留 |
| 规则召回率 | 应触发的硬规则是否全部触发 |
| 布局推荐接受率 | 推荐布局是否被用户接受 |
| 决策可解释率 | 每项选择是否有有效证据 |
| 知识过期率 | 使用的知识是否仍在验证有效期内 |
