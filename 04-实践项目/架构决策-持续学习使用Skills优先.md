# 架构决策：持续学习先使用 Skills 与工作流，再升级为 Agent

状态：已接受  
日期：2026 年 6 月 9 日

## 背景

项目需要持续学习新的组件展示方式、风格和用户偏好。可以选择：

1. 让生产生成 Agent 自己学习和修改知识；
2. 一开始建立多个研究 Agent；
3. 先把学习任务拆成 Skills，由固定工作流和人工评审编排；
4. 完全依赖人工维护。

## 决策

采用“Skills 优先、Agent 后置、生产与学习分离”的方案：

- 生产 Agent 对正式设计知识只读；
- 将覆盖分析、风格研究、变体提议、实现、评测和发布拆成独立 Skills；
- 早期由固定工作流和人工评审编排；
- Skills 边界、数据和评测稳定后，再引入设计研究 Agent 自动规划；
- 正式知识发布始终保留认证和回归门禁。

这里的 Skills 指产品内部可复用、可测试的能力模块，不特指某个具体 Agent 框架的 Skill
实现形式。

## 原因

- 设计研究包含大量主观判断，早期缺少足够数据让 Agent 自主规划；
- Skills 更容易单独测试和改进；
- 固定认证流程可以防止低质量变体污染生产；
- 经过实际运行后，才能知道哪些步骤确实需要 Agent 动态决策；
- 未来可以复用同一组 Skills 给单 Agent 或多 Agent 调用。

## 何时升级为设计研究 Agent

满足以下多数条件时再升级：

- 每周都有稳定的研究和孵化任务；
- Skills 输入输出和失败类型已经清晰；
- 已积累足够用户反馈和认证变体；
- 人工主要时间花在规划和整理，而非审美最终确认；
- 自动规划能带来明确效率收益；
- 有完整评测、预算、权限和回退机制。

## 何时考虑多 Agent

只有在以下情况下考虑拆分研究、实现和评测 Agent：

- 任务可以真正并行；
- 单 Agent 上下文过大，导致质量下降；
- 不同角色需要明显不同权限和模型；
- 独立评测 Agent 能降低自评偏差；
- 多 Agent 增加的复杂度小于收益。

即使使用多 Agent，最终认证仍由人完成。
