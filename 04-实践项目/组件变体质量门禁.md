# 组件变体质量门禁

## 目的

确保组件展示效果不仅丰富，而且可以稳定预览、生成 Monkey C、运行并保持风格一致。

## 变体入库门禁

| 门禁 | 必须满足 |
| --- | --- |
| 来源 | 有明确作者、案例抽象或 Agent 提议记录 |
| 版权与许可 | 图标、字体、位图和案例来源可合法使用 |
| 可实现性 | 只使用受支持绘制原语，有对应 Monkey C 模板 |
| 风格一致性 | 使用指定风格系统的令牌和规则 |
| 状态完整性 | 覆盖要求的数据状态和显示模式 |
| 自动视觉检查 | 边界、重叠、对比度、最小字号全部通过 |
| 设备验证 | 目标设备模拟器或真机验证通过 |
| 性能 | 绘制和资源成本在预算内 |
| 人工审核 | 设计评分达到阈值 |
| 可追溯性 | 有版本、适用范围、预览和验证记录 |

## 评分表

每项 1 至 5 分。任何单项低于 3 分不能认证，总分至少 32 分。

| 维度 | 判断内容 |
| --- | --- |
| 数据表达清晰度 | 用户是否能快速理解组件数值或状态 |
| 风格代表性 | 是否明显符合运动、科技、卡通等目标风格 |
| 状态完整性 | 不同状态是否清晰且保持一致 |
| 小尺寸可读性 | 在目标设备尺寸下是否可识别 |
| 视觉完成度 | 细节、间距、对齐和比例是否成熟 |
| 组合兼容性 | 是否容易与同风格其他组件组合 |
| 设备适配 | Active 与 Always-On 是否合理 |
| 实现一致性 | Web 预览和 Monkey C 结果是否一致 |

## 新变体实验流程

1. Agent 或人工提出变体配方；
2. 使用多组测试数据渲染状态矩阵；
3. 自动检查；
4. 人工评分；
5. 生成 Monkey C 并运行设备验证；
6. 进入小范围候选池；
7. 收集用户选择和拒绝原因；
8. 达到阈值后升级为 `certified`。

## 防止“丰富但混乱”

- 每个候选只能选择一个主风格系统；
- 组件变体必须声明兼容风格；
- 同一候选使用统一字体家族、线宽等级、圆角等级和图标语言；
- 装饰变体数量设置上限；
- 差异计划优先改变布局或表达方式，而不是随机改变所有参数；
- 同一批候选既要不同，也要让用户看出它们属于同一个需求和风格。
