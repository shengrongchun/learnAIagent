# 架构决策：使用共享设计规格驱动预览与 Monkey C

状态：已接受  
日期：2026 年 6 月 9 日

## 背景

产品必须在生成 Monkey C 前让用户确认预览。如果预览由图片模型自由生成，而 Monkey C
由代码模型另行生成，两个结果无法稳定一致。用户确认的效果也无法成为代码验收标准。

## 决策

建立版本化 `WatchFaceSpec` 作为唯一设计事实来源：

1. Agent 将中文需求转换为规格；
2. 网页预览渲染器根据规格绘制效果；
3. 用户反馈修改规格，而不是直接修改图片；
4. 用户确认后锁定规格版本；
5. Monkey C 生成器只根据确认规格生成代码；
6. 最终实现与确认规格做一致性检查。

网页预览用于快速交互，Connect IQ 模拟器或真机渲染用于最终验收。两者出现差异时，以
设备实际效果为事实，并根据差异决定自动修复或返回用户重新确认。

## 为什么不用 AI 图片作为准确预览

- 图片模型不保证组件位置和数据字段准确；
- 图片效果难以转换成确定的坐标、字体和模式逻辑；
- 很难验证 Monkey C 是否忠实实现图片；
- 每次生成存在随机性，不利于回归测试。

AI 图片可以用于早期风格灵感，但不能替代产品中的准确预览。

## 结果

优点：

- 预览与代码可以共享验收标准；
- 设计修改可版本化和回退；
- 可以自动检查边界、重叠和对比度；
- 降低无效代码生成与编译成本；
- 便于支持新的设备和代码模板。

代价：

- 必须认真设计 `WatchFaceSpec`；
- 预览渲染器和 Monkey C 生成器都要支持相同组件；
- 新增设计能力时需要同步扩展两端。

## 约束

- 未被规格表达的视觉能力，第一版不支持；
- Monkey C 自动修复不能擅自改变确认规格；
- 规格变更必须产生新版本；
- 每个最终代码结果必须能追溯到一个确认规格版本。
