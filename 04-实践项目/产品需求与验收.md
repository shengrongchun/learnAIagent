# Garmin 表盘智能生成 Agent：产品需求与验收

## 产品问题

传统 Garmin 表盘开发需要反复完成需求理解、布局尝试、设备适配、Monkey C 编码和编译
调试。该产品的目标是缩短从中文需求到高质量 Monkey C 代码的时间，同时保留用户对视觉
效果的控制。

## 目标用户

- 了解表盘需求但不熟悉 Monkey C 的设计或产品人员；
- 希望快速尝试多个布局的 Garmin 开发者；
- 需要批量制作同一设计系列的开发团队。

## 核心用户故事

### 用户故事 1：从需求获得候选预览

作为用户，我输入中文需求并选择设备后，希望获得多个可比较的表盘预览，以便在写代码前
确认方向。

验收：

- 系统识别缺失信息并提出澄清问题；
- 生成至少 3 个有真实差异的候选；
- 候选使用与目标风格匹配的认证组件视觉变体；
- 候选之间有计划地改变布局或组件表达，而不是随机变色；
- 每个候选展示普通模式与 Always-On 模式；
- 预览尺寸与目标设备分辨率一致；
- 自动标出越界、重叠、低对比度等问题。

### 用户故事 2：通过反馈修改设计

作为用户，我希望用中文反馈调整预览，并保留修改历史。

验收：

- 支持“时间更大、心率移到底部、强调色改为绿色”等反馈；
- 修改只影响相关设计属性；
- 每次修改产生新的规格版本；
- 用户可以回退到之前版本；
- 用户明确确认后才进入代码生成。

### 用户故事 3：从确认预览生成代码

作为用户，我希望系统根据已确认预览生成 Monkey C，而不是重新自由设计。

验收：

- 代码生成引用已确认的规格版本；
- 关键布局、颜色、字段和模式与规格一致；
- 未经确认不得改变设计；
- 生成文件限定在允许目录。

### 用户故事 4：获得通过验证的代码

作为用户，我希望获得至少可以编译，并符合目标设备规则的 Monkey C。

验收：

- 对目标设备执行规则检查和编译；
- 编译失败时最多自动修复指定次数；
- 修复不得改变已确认的视觉设计，若必须改变则返回用户重新确认；
- 输出最终代码，并在系统内部保留可追溯的验证结果；
- 无法通过时明确失败原因，不伪装成功。

### 用户故事 5：确认最终设备效果

作为用户，我希望最终代码在 Connect IQ 模拟器或真机上的效果与确认预览一致。

验收：

- 编译后获取目标设备的实际渲染效果；
- 对比关键组件位置、字号、颜色、字段和模式；
- 差异超出阈值时不得标记为成功；
- 若修复会改变设计，返回预览阶段重新确认；
- 最终代码可追溯到需求、确认规格和验证记录。

正式实现前需要继续完善 JSON Schema，但必须坚持：

- 坐标和尺寸可验证；
- 字段有明确枚举和默认值；
- 规格有版本号；
- 每次用户确认关联唯一规格版本。
- 组件选择、布局、颜色和状态样式能够追溯到知识条目与决策理由。
- 组件视觉变体能够追溯到版本、认证记录和对应 Monkey C 实现。

## 领域知识决策验收

- 每个被展示组件都有选择理由和组件知识引用；
- 每个被展示组件都有明确视觉变体，且与候选主风格兼容；
- 每个被省略的用户可选组件都有省略理由；
- 每个组件至少定义 normal、unavailable 和 Always-On 的适用策略；
- 预览器可以切换关键数据状态与设备显示状态；
- 代码包含运行时状态选择逻辑，而不是只实现一张静态表盘；
- 设备不支持的组件和效果在候选生成前被过滤；
- 布局选择能够说明适用屏幕、组件数量和风格依据；
- 状态颜色使用语义规则，不由模型随意指定；
- 用户反馈能够影响后续候选排序，但不能覆盖硬规则。
- Agent 提议的新视觉变体默认进入实验区，未经认证不得进入正式输出。

## 质量分级

| 等级 | 定义 |
| --- | --- |
| 不合格 | 无法预览、明显违反需求、无法编译或违反设备规则 |
| 可验证 | 预览和编译通过，但设计质量或一致性未达到使用要求 |
| 可用 | 用户确认预览，代码通过验证，少量调整即可使用 |
| 优质 | 设计、设备实机效果、兼容性、可维护性和效率均达到稳定标准 |

项目 v1.0 的最低目标是：核心范围内大部分结果达到“可用”，而不只是“可验证”。

## 评测数据集

至少建立 30 条真实需求，覆盖：

- 简单、详细和信息缺失的需求；
- 不同目标设备；
- 不同字段优先级和视觉风格；
- 普通模式与 Always-On；
- 相互冲突或无法实现的需求；
- 修改反馈和版本回退；
- 会导致编译错误或设备规则冲突的情况。

每条数据记录：

- 原始需求和澄清过程；
- 目标规格；
- 候选预览评分与用户选择；
- 确认规格版本；
- 编译和修复记录；
- 最终人工质量等级。

组件视觉变体需要单独建立测试集，覆盖：

- 不同风格系统下的同一组件；
- 同一风格下多个组件变体；
- 每个变体的关键数据状态；
- Active 与 Always-On；
- Web 预览与 Monkey C 实际效果；
- 单组件评分和组合后评分；
- 用户选择、拒绝和修改原因。
