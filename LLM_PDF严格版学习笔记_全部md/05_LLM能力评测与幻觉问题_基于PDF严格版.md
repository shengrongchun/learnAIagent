## 2. 为什么需要评测？

PDF 说，LLM 展现出强能力，但底层原则仍不完全清楚，并且可能产生：

```text
toxic 有毒内容
biased 偏见内容
harmful 有害内容
fictitious 虚构内容
hallucinations 幻觉
factual errors 事实错误
```

小白解释：

```text
模型很强，但不能完全相信。
```

所以我们需要评测：

```text
它到底会不会做？
做得多稳定？
在哪些任务上容易错？
有没有安全风险？
有没有编造？
```

---

## 3. PDF 中的 Capacity Evaluation 是什么？

PDF 把 capacity evaluation 作为 LLM 综述的四大方面之一。

四大方面是：

```text
1. Pre-training
2. Adaptation
3. Utilization
4. Capacity Evaluation
```

Capacity Evaluation 可以理解为：

```text
评估大模型有哪些能力，以及这些能力表现到什么程度。
```

小白解释：

```text
不要只看模型演示。
要用任务和指标验证模型到底行不行。
```

---

## 4. 为什么 LLM 的评测很难？

基于 PDF 内容，可以总结出几个原因：

```text
1. LLM 能做的任务范围很广。
2. 有些能力是涌现的，很难提前预测。
3. 语言模型损失下降，不一定代表具体任务一定变好。
4. 不同任务的评价指标不同。
5. 开放式回答很难自动判断好坏。
6. 模型可能给出看似合理但实际错误的答案。
7. 安全和对齐问题需要额外评估。
```

小白解释：

```text
评测 LLM 不是做一张选择题卷子那么简单。
因为它能写文章、写代码、聊天、推理、总结、翻译，还可能调用工具。
```

---

## 5. 评测什么能力？

PDF 中讨论 LLM 的能力时，涉及多类能力。结合全文可整理为：

```text
语言理解
文本生成
上下文学习
指令跟随
逐步推理
代码能力
数学能力
知识问答
多轮对话
安全对齐
工具使用
多模态能力
```

小白解释：

```text
不同模型强项不一样。
有的擅长代码，有的擅长中文，有的擅长推理，有的擅长多模态。
```

---

## 6. 涌现能力的评测问题

PDF 第 5～6 页说，涌现能力是小模型没有、大模型出现的能力。  
但 PDF 也提到，对涌现能力的合理性存在讨论，有观点认为部分涌现现象可能与评测设置有关，例如不连续的评价指标。

小白解释：

```text
有些能力看起来像“突然出现”，
但也可能和我们怎么测有关。
```

例如：

```text
如果指标只有“对/错”，
模型从 49% 到 51% 可能看起来像突然跨过门槛。
```

但对真实用户来说，很多任务确实是非黑即白的：

```text
代码能不能运行
答案是否正确
工具是否调用成功
```

所以涌现能力仍然对实际使用很重要。

---

## 7. 幻觉 Hallucination 是什么？

PDF 多处提到 LLM 可能生成 fictitious content、hallucinations 或 factual errors。

小白解释：

```text
幻觉 = 模型编造看起来很真实但实际不可靠的内容。
```

例如：

```text
编造不存在的论文
编造不存在的 API
把错误事实说得很肯定
给出没有依据的结论
```

---

## 8. 幻觉为什么危险？

因为 LLM 的输出经常非常流畅、自然、像真的。

小白解释：

```text
模型说得像真的，不代表是真的。
```

对普通聊天来说，幻觉可能只是误导。  
对业务系统来说，幻觉可能导致：

```text
错误决策
错误代码
错误配置
错误操作
安全风险
用户信任下降
```

---

## 9. GPT-4 相关安全问题

PDF 第 10～11 页讨论 GPT-4 时提到：

```text
GPT-4 更强，但仍存在限制，例如 hallucinations、privacy、overreliance 等。
OpenAI 采用了安全开发和迭代部署策略。
```

小白解释：

```text
越强的模型越需要安全评估。
因为它能做更多事情，也可能造成更大影响。
```

几个关键词：

```text
hallucination：幻觉
privacy：隐私
overreliance：过度依赖
red teaming：红队测试，用来发现风险和漏洞
iterative deployment：迭代部署，逐步发布和降低风险
```

---

## 10. 评测方式：自动评测与人工评测

PDF 中提到开放式评测可以提高评估准确性，但会带来更高成本，因为可能需要人类或 LLM 参与。

小白解释：

```text
选择题、固定答案任务容易自动评测。
开放式回答、设计方案、复杂推理很难自动评测。
```

例如：

```text
数学题：
有标准答案，容易评测。

开放式方案：
答案可能有很多种，需要人或更强模型判断质量。
```

---

## 11. Benchmark 基准测试

PDF 在能力评测部分和多模态部分提到很多 benchmark。Benchmark 可以理解为：

```text
一套固定测试题和评价规则。
```

作用：

```text
比较不同模型
追踪模型版本变化
发现模型短板
评估某类能力
```

小白解释：

```text
Benchmark 就像考试卷。
不同考试卷测不同能力。
```

但要注意：

```text
Benchmark 分数高，不代表真实业务一定好。
真实业务仍然需要自己的评测集。
```

---

## 12. 为什么业务系统需要自己的评测集？

PDF 讨论的是通用 LLM 能力评测。  
但真实应用中，通用 benchmark 不一定覆盖你的业务场景。

小白解释：

```text
模型在公开考试中高分，
不代表它在你的业务任务里稳定。
```

比如你的业务可能更关心：

```text
是否按格式输出
是否调用正确工具
是否引用正确资料
是否拒绝不该做的操作
是否生成可执行代码
是否产生幻觉
```

这些需要业务自己的测试集。

注意：这一节后半部分属于应用理解，但评测必要性来自 PDF 的 capacity evaluation 思想。

## 13. 常见评测维度

基于 PDF 对能力与风险的讨论，可以整理出常见评测维度：

```text
1. 正确性：答案是否正确
2. 一致性：同类问题是否稳定
3. 指令跟随：是否按要求执行
4. 推理能力：是否能解决多步任务
5. 事实性：是否编造
6. 安全性：是否输出有害内容
7. 鲁棒性：输入变化后是否仍可靠
8. 工具使用：是否正确调用外部工具
9. 成本与延迟：是否可落地
10. 人类满意度：用户是否认可
```

---

## 14. 幻觉缓解思路

PDF 没有把所有工程方案整理成一个表，但根据全文内容，相关思路包括：

```text
1. Alignment tuning：让模型更符合人类偏好和安全要求
2. RAG：用检索资料增强事实依据
3. Tools：用外部工具补足模型短板
4. Prompt design：更清晰地约束任务和输出
5. Self-check / confidence assessment：让模型检查生成质量或判断是否需要检索
6. Evaluation：用评测集发现问题
7. Red teaming：主动测试风险场景
```

小白解释：

```text
减少幻觉不能只靠一句“不要胡说”。
要靠资料、工具、校验、评测和安全机制一起配合。
```

## 15. 本章小白总结

```text
1. LLM 很强，但会错，也会编造。
2. 评测是判断模型是否可靠的关键。
3. 通用 benchmark 能比较模型，但不能代替业务评测。
4. 幻觉是 LLM 的重要风险。
5. GPT-4 这样的强模型仍然需要安全评估。
6. 开放式任务评测更难，可能需要人或 LLM 参与。
7. 真实业务系统必须建立自己的评测集。
8. RAG、工具、对齐、Prompt、自检、红队测试都可以帮助降低风险。
```

## 16. 【延伸应用】和表盘生成系统的关系

> 重要：本节不是 PDF 原文，而是基于 PDF 中 capacity evaluation、hallucination、tools、RAG、安全等概念，结合你的 Garmin 表盘生成系统做的应用理解。

---

### 16.1 表盘系统为什么需要评测？

因为模型可能：

```text
生成格式错误的 JSON
编造不存在的组件 ID
编造不存在的 Monkey C API
生成视觉上重叠的布局
声称可实现但实际不可实现
忽略 low_power / ambient 状态
```

这些属于表盘业务中的“幻觉”或“不可靠输出”。

---

### 16.2 表盘系统评测指标

可以设计：

```text
1. JSON 合法率
2. Schema 校验通过率
3. 组件存在率
4. Monkey C 可实现率
5. 布局不重叠率
6. 圆形安全区域通过率
7. normal / low_power / ambient 状态完整率
8. SVG 渲染成功率
9. 人工美观评分
10. 修改后通过率
```

这些指标不是 PDF 原文，是应用设计。

---

### 16.3 表盘系统中的幻觉例子

```text
模型输出 type = glassPanel，但渲染器不支持。
模型说可以用 drawBlur，但 Garmin Monkey C 没有这个能力。
模型编造 component_id = luxury_heart_99，但组件库不存在。
模型生成坐标超出 416x416。
模型把多个组件放在同一区域导致重叠。
```

---

### 16.4 表盘系统的安全边界

结合 PDF 的安全与对齐思想，你的系统应该：

```text
先生成方案，不直接覆盖文件
先生成预览，不直接发布
先检查 Monkey C 可实现性，再生成代码
高风险操作需要人工确认
所有失败案例进入评测集
```
