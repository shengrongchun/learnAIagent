# 项目驱动的 AI Agent 90 天路线图

计划周期：2026 年 6 月 10 日至 2026 年 9 月 9 日  
投入：每天约 2 小时，每周约 14 小时  
真实产品项目：Garmin 表盘智能生成 Agent

## 学习目标

Garmin 项目必须作为真实产品交付，同时承担学习载体的作用。每周必须同时获得两类成果：

1. **项目成果**：代码生成 Agent 增加一个可演示、可验证的能力；
2. **通用 Agent 能力**：能够脱离 Garmin 场景解释原理、选型和失败模式。

三个月后的目标不是自称专家，而是达到“具备专家成长路径、能独立负责小型 Agent
项目”的水平。成为真正专家还需要持续做多个项目、处理线上问题并形成方法论。

## 总体路线

| 阶段 | Agent 主线 | Garmin 项目成果 |
| --- | --- | --- |
| 第 1 至 3 周 | 模型、结构化输出、领域建模与混合检索 | 中文需求生成有依据的设计规格 |
| 第 4 至 5 周 | 预览、人在回路、RAG 和质量门禁 | 高质量候选预览与规格锁定 |
| 第 6 至 8 周 | 代码生成、工具调用、Agent 循环、MCP | 编译、诊断、修复和一致性验证 |
| 第 9 至 10 周 | 产品评测、安全、可观测性 | 批量生成质量闭环 |
| 第 11 至 13 周 | 持续学习、架构、产品与负责人能力 | 双循环产品、答辩和迁移设计 |

## 每周固定学习方法

- 周一：理解一个 Agent 概念，写出自己的解释；
- 周二：不使用框架完成最小实验；
- 周三至周四：把能力加入 Garmin 项目；
- 周五：建立测试集，收集失败案例；
- 周六：修复、重构并记录数据；
- 周日：演示、复盘，并说明如何迁移到其他场景。

从第 3 周开始，每周额外运行一次小型设计学习循环：

1. 分析本周用户反馈和设计覆盖缺口；
2. 选择一个值得研究的组件或风格问题；
3. 提议并预览一个实验变体；
4. 记录评测和人工判断；
5. 决定继续实验、认证或淘汰。

## 第 1 周：Agent 全景与业务问题

Agent 能力：

- 区分聊天机器人、工作流和 Agent；
- 理解模型、Prompt、RAG、工具、记忆、评测的位置；
- 从业务目标判断是否应该使用 Agent。

项目实践：

- 分析 Garmin 代码生成的用户、输入、输出、约束和风险；
- 手工写 5 条表盘需求及对应 Monkey C 代码要求；
- 画出第一版 Agent 架构图。

验收：

- 能解释为什么该项目需要 Agent，以及哪些步骤不应该交给 Agent；
- 能把同一套分析方法用于“客服 Agent”或“数据分析 Agent”。

## 第 2 周：模型 API 与结构化输出

Agent 能力：

- 模型 API、角色消息、上下文、温度和 Token；
- System Prompt、Few-shot、结构化 JSON 输出；
- Schema 校验、错误处理和重试。

项目实践：

- Node.js 调用模型 API；
- 把中文表盘需求转换为结构化规格；
- 建立 20 条需求样例，比较 3 版提示词。

验收：

- 规格合法率有数据；
- 能解释提示词为什么有效以及何时会失效。

## 第 3 周：领域建模与设计决策

Agent 能力：

- 结构化知识、规则引擎、语义检索和 Agent 决策的边界；
- 领域实体、状态、风格系统、组件视觉变体、规则和决策证据；
- 如何构建可解释的推荐与选择。

项目实践：

- 建立首批设备、时间、日期、心率、电量和步数组件知识；
- 为组件定义 normal、unavailable、warning 和 Always-On 状态；
- 定义运动风、科技风、卡通风的设计令牌和组件选择规则；
- 为核心组件建立首批视觉变体和质量门禁；
- 定义覆盖分析、风格研究、变体提议和评测 Skills；
- 建立 3 个布局模式、语义色和关键硬规则；
- 实现组件可行性过滤、认证变体查询、评分和布局选择；
- 输出带 `DecisionTrace` 的 `WatchFaceSpec`。

验收：

- 能解释每个组件为什么被选择或省略；
- 能解释每种状态、颜色、效果和组件变体的依据；
- 能说明丰富性如何来自受控组合，而不是随机生成；
- 模型无法绕过设备与硬规则；
- 能说明为什么领域知识不能只放入向量库。
- 能说明 Skill、工作流和 Agent 的边界。

## 第 4 周：共享规格、预览、人在回路与 RAG

Agent 能力：

- 上下文选择、示例选择、模板和约束；
- 文档加载、切分、Embedding、语义检索和引用；
- Human in the Loop、状态和版本管理；
- 检索失败、决策失败、预览失败与生成失败的区别。

项目实践：

- 使用 Vue + SVG/Canvas 根据 `WatchFaceSpec` 渲染准确预览；
- 预览可切换组件数据状态和设备显示模式；
- 预览可以展示不同风格和组件视觉变体；
- 为模糊风格和优秀设计案例加入语义检索；
- 使用候选差异计划生成多个丰富但风格一致的预览；
- 支持中文修改反馈、版本记录和用户确认。

验收：

- 用户可以在写 Monkey C 前完成预览选择和修改；
- 预览与后续代码生成共享同一份规格；
- 设计决策和语义检索结果能够引用知识来源；
- 正式候选只使用认证组件变体；
- 能说明人在回路为什么是产品质量机制；
- 能把 RAG 方法迁移到企业知识库 Agent。

## 第 5 周：设计质量与约束驱动生成

Agent 能力：

- 结构化约束、规则引擎、Guardrail 和视觉质量门禁；
- 模型判断与程序校验如何配合；
- 失败分类和回退策略。

项目实践：

- 检查布局越界、元素重叠、对比度、字段优先级和 Always-On；
- 检查组件变体质量、组合风格一致性和候选差异有效性；
- 建立预览人工评分表和首轮接受率；
- 对失败设计给出明确原因并定向修改；
- 对比“只有 Prompt”和“Prompt + 规则检查 + 人工确认”的结果。

验收：

- 用数据证明质量门禁提高了预览接受率；
- 能说明 Guardrail 与安全策略的通用设计。

## 第 6 周：确认规格到 Monkey C

Agent 能力：

- Function Calling / Tool Calling；
- 工具描述、参数 Schema、超时、重试和权限；
- 工作流适合固定步骤，Agent 适合不确定决策。

项目实践：

- 固定 Connect IQ 项目骨架和可复用代码模板；
- 仅根据用户确认的规格生成 Monkey C；
- 实现设备查询、代码写入、规则检查和编译工具；
- 先用固定工作流串联规格、代码与工具；
- 记录每次工具调用的输入、输出和错误。

验收：

- 代码忠实实现已确认规格，不重新自由设计；
- 工具错误不会导致整个程序崩溃；
- 能解释什么时候应该使用工作流而不是 Agent。

## 第 7 周：Agent 循环与自动修复

Agent 能力：

- 目标、规划、行动、观察和结束条件；
- 最大步数、循环检测、错误恢复；
- 状态管理和短期记忆。

项目实践：

- 手写“生成 -> 编译 -> 分析错误 -> 修复 -> 再编译”循环；
- 设置最大修复次数和失败退出；
- 测试 10 个编译失败案例。

验收：

- 可以追踪每次修复为什么发生；
- Agent 不会无限循环；
- 能把该模式迁移到代码修复 Agent 或运维 Agent。

## 第 8 周：Agent 框架、MCP 与架构判断

Agent 能力：

- LangChain / LangGraph 的价值和代价；
- MCP 客户端、服务器、资源和工具；
- 框架选型与避免过度设计。

项目实践：

- 将一个构建或规则查询能力封装成 MCP 工具；
- 用 Agent 框架重构一段流程；
- 加入最终代码与确认预览的一致性检查；
- 对比手写版和框架版的代码量、可控性和可观测性。

验收：

- 能有依据地决定是否使用框架和 MCP；
- 视觉实现差异超过阈值时不能输出为成功；
- 不以“使用了热门框架”作为项目价值。

## 第 9 周：Agent 评测

Agent 能力：

- 离线评测集、回归测试和人工评测；
- 任务成功率、正确性、成本、延迟和步骤数；
- 模型评审的偏差和局限。

项目实践：

- 建立至少 30 条真实表盘需求评测集；
- 自动运行需求完整性、预览规则、规格一致性、代码规则和编译评测；
- 加入人工预览评分、首轮接受率和最终可用率；
- 每次修改前后生成对比报告。

验收：

- 能用数据判断 Agent 是否变好；
- 能设计另一个 Agent 场景的基础评测方案。

## 第 10 周：安全与可观测性

Agent 能力：

- Prompt Injection、越权工具调用和数据泄露；
- 日志、Trace、告警、预算、降级和人工接管；
- 生产 Agent 的责任边界。

项目实践：

- 限定文件写入目录和允许执行的工具；
- 添加请求 ID、结构化日志和成本记录；
- 编写 10 条恶意需求或越权测试；
- 模型不可用或反复失败时安全退出。

验收：

- 能复现一次任务的完整执行过程；
- 能说明上线一个 Agent 前必须完成的安全检查。

## 第 11 周：从单 Agent 到系统设计

Agent 能力：

- 单 Agent、工作流和多 Agent 的取舍；
- Skills、工作流、Agent 和多 Agent 的职责边界；
- 模型路由、任务队列、并发和缓存；
- 质量、成本、速度之间的权衡。

项目实践：

- 支持批量生成任务，并限制并发和预算；
- 实现设计覆盖分析、变体提议、自动评测和知识发布等孵化 Skills；
- 建立“生产生成循环 + 设计孵化循环”；
- 分析何时将孵化工作流升级为设计研究 Agent；
- 分析是否需要多 Agent，默认优先 Skills + 工作流或单 Agent + 工具；
- 编写架构决策记录。

验收：

- 能拒绝没有收益的多 Agent 设计；
- 能解释生产 Agent 为什么不能未经审核自我学习；
- 能展示一个新变体从缺口发现到认证发布的流程；
- 能解释系统扩展到 1000 个任务时的瓶颈。

## 第 12 周：负责人能力与作品集

Agent 能力：

- 业务价值、范围、里程碑和风险；
- 模型与框架选型、Build vs Buy；
- 如何向业务和技术人员汇报。

项目实践：

- 清理项目并补全 README；
- 整理产品需求、架构图、评测报告、成本和失败复盘；
- 录制 5 至 10 分钟中文演示；
- 完成一次 30 分钟模拟答辩。

验收：

- 能说明为什么值得做、如何证明有效、失败时怎么办；
- 新用户可以按 README 完成“需求 -> 预览确认 -> Monkey C 输出”的核心流程；
- 核心范围内大部分结果达到“可用”，而不只是“能编译”。

## 第 13 周：迁移能力与下一阶段

项目实践：

- 重新完成能力自测；
- 不写代码，为另一个领域设计 Agent 方案；
- 说明哪些模块可复用，哪些领域规则必须重做；
- 制定未来 6 个月专家成长计划。

最终验收：

- 能独立设计一个不同领域的 Agent；
- 能给出架构、评测、安全、成本和迭代方案；
- 不依赖 Garmin 场景解释 Agent 的核心方法。
